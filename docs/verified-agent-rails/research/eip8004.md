# ERC-8004 & the Trustless-Agent EIP Stack — VAR Research

_Consolidated identity / mandate / gating research for Verified Agent Rails: what's real, what's fabricated, and what VAR must build itself._

> **Feasibility verdict: FEASIBLE (partial) — build the gate, don't trust a phantom standard.** VAR's core thesis (one on-chain VIEW call gates every transfer; World ID anchors proof-of-human; revoke → next block the agent is dead) is technically sound and composes **real, shippable** standards: **ERC-7943 (Final)** for the `canTransfer` hook, **ERC-8004 (Draft, live mainnet)** for agent identity, **ERC-8126 (status disputed — see §2)** for optional risk scoring. **The single load-bearing caveat: ERC-8226 ("Regulated Agent Mandate Standard" / RAMS) — the standard the contracts doc names for `isActiveForAmount` and the Mandate struct — could NOT be found in any public spec (Rounds 4 & 5, high confidence). Treat the ERC-8226 *interface shape* as VAR's own design, implemented by a custom `EligibilityResolver` contract — NOT as an external standard you can import or rely on.** With that reframe, there are no hard blockers for the hot path.

**The composition chain (one chain, all views):**

```
GatedToken.transfer(to, amount)
  └─ canTransfer(from, to, amount)              [ERC-7943, Final — view]
       └─ resolver.isActiveForAmount(agentId|agentWallet, principal, amount)  [VAR-OWNED — view]
            ├─ Mandate checks: revoked / expiry / cap / asset whitelist
            └─ agentBook.resolveHuman(agentWallet) != 0   [World-ID anchor — view]
```

Every hop is a `view`. Revoke flips `mandate.revoked = true` in one tx; next block `isActiveForAmount` returns false; the transfer reverts. No paymaster, no relayer, no off-chain attestation on the hot path. **"Accountability with one click."**

### 3.2 ERC-8004 Identity Registry — interface (high confidence)

```solidity
// agentId is the IMMUTABLE ERC-721 tokenId. agentWallet is MUTABLE metadata.
function register(string agentURI, MetadataEntry[] calldata metadata) external returns (uint256 agentId);
function register(string agentURI) external returns (uint256 agentId);
function register() external returns (uint256 agentId);

function getAgentWallet(uint256 agentId) external view returns (address);
function setAgentWallet(uint256 agentId, address newWallet, uint256 deadline, bytes calldata signature) external; // EIP-712 (EOA) / ERC-1271 (contract wallet)
function unsetAgentWallet(uint256 agentId) external;

function getAgentURI(uint256 agentId) external view returns (string); // via ERC-721 tokenURI
function setAgentURI(uint256 agentId, string newURI) external;

// Reputation Registry
function giveFeedback(uint256 agentId, int128 value, uint8 valueDecimals, string tag1, string tag2, string endpoint, string feedbackURI, bytes32 feedbackHash) external;

```

**Global agent identifier schema:** `agentRegistry = {namespace}:{chainId}:{identityRegistry}` (e.g. `eip155:1:0x8004A169…`) **+** `agentId` (uint256 tokenId).

### 3.3 ⚠️ Wallet-rotation trap (drives VAR's keying decision)

```text
When the agent NFT is transferred to a new owner:
  → agentWallet metadata AUTO-CLEARS to address(0)
  → new owner MUST re-verify control via setAgentWallet(agentId, newWallet, deadline, signature)
  → any permission keyed to the old agentWallet is now broken
```

This is by design (prevents stale payment associations crossing ownership). It is the reason the keying decision below matters.

### 3.6 VAR-OWNED EligibilityResolver (the "ERC-8226 shape" — VAR's contract, not a standard)

```solidity
// Interface shape VAR implements itself. NOT an importable standard. See §2 🚩.
function isActiveForAmount(uint256 agentId, address principal, uint256 amount) external view returns (bool);
function revokeMandate(uint256 agentId, address principal) external;   // the kill switch (WRITE)
function grantMandate(/* principal, agentId/agentWallet, spendCap, assetWhitelist, expiry */) external;
function getMandate(uint256 agentId, address principal) external view returns (Mandate memory);

// Mandate struct — VAR's design. The on-chain `revoked` bool is what makes the climax atomic.
struct Mandate {
    address human;            // verified principal (proof-of-human anchor)
    address agent;            // Dynamic-provisioned agent wallet
    uint256 spendCap;         // 6-decimal USDC units (Arc ERC-20 interface)
    uint256 spent;            // cumulative — only updated by recordExecution (WRITE)
    address[] assetWhitelist; // permitted tokens; empty = none
    uint64  expiry;           // unix seconds
    bool    revoked;          // human's kill switch
}
```

> Note: the research bundle's `isActiveForAmount(agentId, principal, amount)` signature carries **no `asset` param** — the asset/whitelist check is VAR-added. Either bind the resolver to one token or extend the signature. Your call; VAR owns this interface.

### 3.7 World ID on-chain verification (seed-time only — never on hot path)

```solidity
function verifyProof(
    uint256 root,
    uint256 groupId,              // 1 = Orb (biometric); legacy creds won't register
    uint256 signalHash,           // keccak256(signal); bind signal = agent wallet address
    uint256 nullifierHash,        // MUST track in a mapping — verifyProof does NOT prevent replay
    uint256 externalNullifierHash,// app-context binding (document VAR's value explicitly)
    uint256[8] calldata proof
) external view;
```

⚠️ `verifyProof` is a **view that does not dedupe** — your seeding contract must store seen `nullifierHash` to enforce one-human-one-anchor. Proof validity has ~5-min L2 Merkle-root propagation latency. This runs **once at registration**, never per transfer.

## 4. ERC-1404-style revert reasons (matches `03_CONTRACTS.md` §4)

```solidity
function detectTransferRestriction(address from, address to, uint256 amount) external view returns (uint8 code);
function messageForTransferRestriction(uint8 code) external pure returns (string memory);
```

| Code | Constant | Triggered when |
|------|----------|----------------|
| 0 | `OK` | all checks pass |
| 1 | `NO_PASSPORT` | no Mandate for the agent wallet |
| 2 | `MANDATE_REVOKED` | `mandate.revoked == true` |
| 3 | `MANDATE_EXPIRED` | `block.timestamp >= mandate.expiry` |
| 4 | `OVER_SPEND_CAP` | `mandate.spent + amount > mandate.spendCap` |
| 5 | `ASSET_NOT_WHITELISTED` | token not in `assetWhitelist` |
| 6 | `NOT_VERIFIED_HUMAN` | `agentBook.resolveHuman(agent) == address(0)` |

Demo arc: first "rejected, no passport" beat = code `1`; kill-shot after revoke = code `2`. Agent should call `detectTransferRestriction` (view) **before** transferring so it never burns gas on a doomed tx.

## 5. What this unblocks for the build

1. **Gate hook is real and Final.** Build `GatedToken` against **ERC-7943** `canTransfer(from,to,amount)→bool`, ERC-20 shape, today. No churn risk.
2. **Identity is live.** Use **ERC-8004** `getAgentWallet`/`setAgentWallet` for agent resolution and (optionally) Reputation `getSummary` for discovery. Don't assume a deployed mandate registry — there isn't one.
3. **Write your own resolver — don't wait for a standard.** The Mandate struct + `isActiveForAmount` + on-chain `revoked` is **VAR's contract** (§3.6). This is the load-bearing keystone and is fully buildable in Solidity in hour 0. P0.
4. **Hot path = one VIEW call, confirmed.** `canTransfer → isActiveForAmount → (mandate checks + agentBook anchor)`, all views. Atomic next-block revocation via `mandate.revoked`. Satisfies "zero new trust assumptions" — eligibility bottoms out in World ID proof-of-human, not a new authority.
5. **Keying decision is locked (MVP): key the Mandate by AGENT WALLET ADDRESS, not agentId.** Rationale: matches the live World ID Agent Passport pattern; avoids agentId↔registry lookups on the hot path; keeps the resolver self-contained. **Accept the limitation:** rotating the agent NFT clears `agentWallet` → human must re-grant. Document this explicitly. (agentId survives transfer but forces an ERC-8004 registry read every transfer — not worth it for 36h.)
6. **Per-tx spend caps only (MVP).** Cumulative caps require a post-transfer `recordExecution` **write**, which breaks the "pure VIEW hot path" invariant. Ship per-tx caps; note cumulative as a stretch with the recordExecution gotcha.
7. **Single chain, hour-0 decision.** Host GatedToken + EligibilityResolver + AgentBook mirror **all on Arc**. Deploy a **local AgentBook mirror** seeded once from a World ID proof (track `nullifierHash`) rather than cross-chain querying World's Base/World-Chain AgentBook. Cross-chain = no local VIEW to read = broken hot path (this is Risk #1 in `08_RISKS.md`).
8. **Proof-of-human is VAR's novel layer.** None of ERC-7943 / 8004 / 8126 define a proof-of-human primitive — they delegate eligibility to an external compliance provider. VAR supplies it via World ID → AgentBook. **Do NOT claim the EIPs mandate proof-of-human; VAR composes it in.**
9. **Payments are orthogonal.** x402 + USDC on Arc settle payments; this is separate from the mandate gate. AP2 mandates key to the **human** signing key (not agent), so AP2 is a different concern — don't over-couple.
