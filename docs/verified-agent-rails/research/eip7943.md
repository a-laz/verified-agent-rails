# ERC-7943 (uRWA) — Token-Level Transfer Gating for Verified Agent Rails

*The keystone standard: a single VIEW-only `canTransfer` hook that lets the asset itself enforce the agent's leash.*

**Feasibility verdict: FEASIBLE (high confidence).** ERC-7943 is **Final** (May 27, 2026). Its fungible `canTransfer(from, to, amount)` is a pure VIEW that MUST NOT revert and MUST NOT write storage — exactly the hot-path primitive VAR needs. VAR's "one VIEW call" thesis holds **for per-transaction caps**. The one hard correction from the research: **ERC-8226 does not exist** (phantom standard cited in early rounds). VAR must ship its own `EligibilityResolver` exposing `isActiveForAmount(...)` — this is the correct, spec-compliant path and is already the locked decision. Cumulative caps (which would require a state write) are correctly deferred to STRETCH so the MVP hot path stays pure VIEW.

### Core interface (ERC-20 / fungible) — all Final

```solidity
interface IERC7943Fungible {
    // Hot-path gate — VIEW, MUST NOT revert, MUST NOT write storage
    function canTransfer(address from, address to, uint256 amount) external view returns (bool allowed);

    // Companion eligibility views
    function canSend(address account)    external view returns (bool allowed);
    function canReceive(address account) external view returns (bool allowed);
    function getFrozenTokens(address account) external view returns (uint256 amount);

    // Regulator / enforcer tooling — state-mutating, OFF the hot path
    function forcedTransfer(address from, address to, uint256 amount) external; // returns bool in some impls
    function setFrozenTokens(address account, uint256 amount) external;          // overwrites (approve-style)
}
```

### Overloaded `canTransfer` across variants (for reference; VAR uses fungible only)

## 3. The VAR composition chain (locked architecture)

The whole hot path is **on-chain VIEW only, co-located on Arc, one revert path**. No paymaster, no relayer, no off-chain attestation, zero cross-chain calls.

```
GatedToken.transfer(to, amount)
  └─ _update(from, to, amount)                                   [ERC-7943 enforcement wrapper]
       ├─ canTransfer(from, to, amount)                          [ERC-7943, VIEW, returns bool]
       │     └─ EligibilityResolver.isActiveForAmount(           [VAR contract, VIEW] (NOT ERC-8226)
       │            agentId, principal, amount)
       │            // checks: mandate exists, !revoked, !expired,
       │            //         spent+amount <= spendCap (per-tx in MVP),
       │            //         asset whitelisted
       │            └─ AgentBook.resolveHuman(agentWallet)       [World-ID-seeded mirror on Arc, VIEW]
       │                   return human != 0 && human == m.human
       └─ if !allowed: read detectTransferRestriction(...) for code, then REVERT(code, message)
```

**Why this is "zero new trust assumptions":** eligibility bottoms out in `AgentBook`, which is seeded **once, off the hot path**, by a World ID proof-of-human (`verifyProof`). The hot path only *reads* that mirror. Revocation is a single mandate state flip (`revoked = true`) → next block `isActiveForAmount` returns false → every transfer reverts. "Accountability with one click."

### The audit-critical revert pattern (do not get this wrong)

ERC-7943 Final mandates `canTransfer` **never reverts**. Therefore:

```solidity
// SAFE (VAR + OpenZeppelin pattern):
//   1. canTransfer() and isActiveForAmount() return bool — never revert, never write
//   2. The token's _update() (OZ v5) wrapper calls canTransfer()
//   3. On false, _update() reads detectTransferRestriction() for a machine-readable code
//   4. _update() performs the REVERT with (code, message)
//
// UNSAFE (violates ERC-7943 Final "MUST NOT revert"):
//   - canTransfer() itself calling revert(...)
```

OpenZeppelin's reference confirms the pattern — `canTransfer` is view-only, enforcement lives in the `_update` chain (`ERC20Freezable` + `ERC20Restricted`):

```solidity
function canTransfer(address from, address to, uint256 amount) external view virtual returns (bool) {
    return (amount <= available(from) && canTransact(from) && canTransact(to));
}
```

### 4.2 Mandate struct (VAR canonical, high confidence)

```solidity
struct Mandate {
    address   human;          // World-ID-verified principal (anchor)
    address   agent;          // agent wallet (Dynamic-provisioned)
    uint256   spendCap;       // per-tx cap in MVP
    uint256   spent;          // running total (used only if STRETCH cumulative caps ship)
    address[] assetWhitelist; // MVP: single hardcoded gated ERC-20
    uint64    expiry;         // block.timestamp >= expiry → expired
    bool      revoked;        // the one-click kill switch
}
```

### 4.3 `isActiveForAmount` pseudocode — MVP per-tx, pure VIEW (high confidence)

```solidity
function isActiveForAmount(agentId, principal, amount, asset) returns (bool) {
    Mandate m = mandates[agentId];
    if (m.human == address(0))           return false; // NO_PASSPORT
    if (m.revoked)                       return false; // MANDATE_REVOKED
    if (block.timestamp >= m.expiry)     return false; // MANDATE_EXPIRED
    if (m.spent + amount > m.spendCap)   return false; // OVER_SPEND_CAP (per-tx in MVP; NO state write)
    if (!_inWhitelist(asset, m))         return false; // ASSET_NOT_WHITELISTED
    address human = agentBook.resolveHuman(m.agent);
    if (human == address(0) || human != m.human) return false; // NOT_VERIFIED_HUMAN
    return true; // OK
}
```

### 4.4 ERC-1404-style restriction codes (VAR-local convention, medium confidence)

These are **VAR's**, not part of ERC-7943 (which uses custom errors). Use them for machine-readable pre-flight + revert reasons.

```
0 = OK
1 = NO_PASSPORT            (no mandate found)
2 = MANDATE_REVOKED        (human flipped kill switch)
3 = MANDATE_EXPIRED        (block.timestamp >= expiry)
4 = OVER_SPEND_CAP         (spent + amount > cap)
5 = ASSET_NOT_WHITELISTED  (asset not in whitelist)
6 = NOT_VERIFIED_HUMAN     (agentBook.resolveHuman == 0 or mismatch)
```

Wiring: add `detectTransferRestriction(from, to, amount) view returns (uint8)` and `messageForTransferRestriction(uint8) view returns (string)` to the GatedToken. The agent pre-flights `detectTransferRestriction` (cheap `eth_call`) to avoid wasted gas and to get a clean reason; the actual revert (with the code + message) happens in `_update`.

## 5. What this unblocks for the build

- **GatedToken (Lane A) can start now.** ERC-20 + `IERC7943Fungible`; override OZ v5 `_update` to call `canTransfer`, and on `false` read `detectTransferRestriction` and revert with `(code, message)`. Pattern is locked and spec-compliant.
- **EligibilityResolver can start now.** Implement `isActiveForAmount` exactly as §4.3. **No dependency on any external standard** — do not import or reference ERC-8226.
- **AgentBook mirror lives on Arc, seeded once.** `verifyProof` (VIEW) + on-chain `nullifierHash` mapping. Hot path only reads `resolveHuman(agentWallet)`. No bridge/oracle needed for the demo → kills the cross-chain risk.
- **MVP scope is secured.** Per-transaction caps only → pure VIEW hot path → "one VIEW call" thesis intact. Cumulative caps deferred to STRETCH (they'd need a post-transfer state write that changes the narrative).
- **Settlement layer is go.** Arc testnet live; USDC at `0x3600…` via 6-decimal ERC-20 interface; x402/Nanopayments available for the sub-cent agent payment loop.
- **Yield step is de-risked.** MVP uses an ERC-4626 stub (1:1, no APY); real yield is the most-cuttable STRETCH item.
- **Revocation demo is trivial and atomic.** Human flips `mandate.revoked = true` from phone → next block every transfer reverts. No agent code change.
