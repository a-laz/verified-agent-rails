# ERC-8226 — Regulated Agent Mandate Standard (RAMS)

*The eligibility resolver that turns VAR's `canTransfer` into a one-VIEW-call accountability gate.*

**Feasibility verdict: GREEN — `feasible: yes`.** ERC-8226 is a **real, public, merged Draft** Ethereum standard (ethereum/ERCs PR #1679, merged 2026-05-12, commit `3edf21b`). The hot-path signature `isActiveForAmount(uint256 agentId, address principal, uint256 amount) → bool` is **canonical and locked**. VAR's critical-path spine (`token → canTransfer → isActiveForAmount → AgentBook`) can pin directly to the published `IAgentMandate` interface with no guessing. One implementation constraint survives: **`isActiveForAmount` carries NO asset parameter** — asset whitelisting moves into the token's `canTransfer`. Resolve that and Lane A ships.

## 1. What this is and where it sits

ERC-8226 ("Regulated Agent Mandate Standard" / RAMS) is a registry standard for delegating **financial/jurisdictional authority** from a human **principal** to an autonomous **agent**, with on-chain spend caps, expiry, asset scope, and revocation. It is the "who authorized this agent and with what caps" layer.

```
VAR compliance spine (hot path = ONE view call):

  GatedToken (ERC-7943, Final)
     └─ canTransfer(from, to, amount)               [token-level investor eligibility]
          └─ RAMS registry (ERC-8226 IAgentMandate, Draft)
               ├─ getActivePrincipal(agentId) → principal   [resolve agent → human]
               └─ isActiveForAmount(agentId, principal, amount) → bool   [mandate caps]
                    └─ AgentBook (World ID-seeded)            [proof-of-unique-human anchor]
```

**Standards delineation (do not conflate):**

| Standard | Status | Owns | VAR role |
|----------|--------|------|----------|
| **ERC-7943** (uRWA) | **Final** (May 2026) | `canTransfer` / `canSend` / `canReceive` token gate | GatedToken extends this |
| **ERC-8226** (RAMS) | **Draft** (May 2026) | Mandate lifecycle, caps, revocation, principal resolution | EligibilityResolver / AgentBook pins to `IAgentMandate` |
| **ERC-8004** | Live (Jan 2026) | Agent identity / reputation / validation (ERC-721 agentId) | resolve agent wallet → `agentId` |
| **ERC-8118** | Draft (PR #1450) | "What functions can the agent call" (capability delegation) | ⚠️ out of VAR MVP scope — see §9 |

ERC-7943 does **not** name ERC-8226 in its spec; composition is implicit and vendor-agnostic via the token implementation. ERC-8004 and ERC-8226 are separate, complementary trust dimensions (identity vs. compliance).

## 2. Canonical interface (pin to these — high confidence)

All signatures below are extracted from `raw.githubusercontent.com/ethereum/ERCs/master/ERCS/erc-8226.md` and cross-checked against `eips.ethereum.org/EIPS/eip-8226` (HTTP 200). Both interfaces implement ERC-165.

### 2.1 `IAgentMandate` — the registry VAR's AgentBook implements/wraps

```solidity
interface IAgentMandate {
    // ── hot path (VIEW) ──────────────────────────────────────────
    function isActiveForAmount(uint256 agentId, address principal, uint256 amount)
        external view returns (bool);
    function getActivePrincipal(uint256 agentId) external view returns (address);
    function isActive(uint256 agentId) external view returns (bool);          // ⚠️ verify exact sig
    function getMandate(uint256 agentId, address principal)
        external view returns (Mandate memory);                              // ⚠️ verify exact sig
    function isFrozen(uint256 agentId) external view returns (bool);          // ⚠️ verify exact sig
    function isOperator(address operator) external view returns (bool);       // ⚠️ verify exact sig

    // ── write path ───────────────────────────────────────────────
    function grantMandate(
        uint256 agentId,
        address principal,
        bytes32 identityRef,
        bytes32 scopeHash,
        MandateScopeParams calldata onChainScope,
        address complianceProvider,
        uint48 validFrom,
        uint48 validUntil,
        bytes calldata signature           // empty if msg.sender == principal
    ) external;
    function revokeMandate(uint256 agentId, address principal) external;
    function extendMandate(/* … */) external;                                // ⚠️ verify exact sig
    function recordExecution(uint256 agentId, address principal, uint256 amount) external;
    function freezeAgent(uint256 agentId, bytes32 jurisdictionHash) external;
    function unfreezeAgent(uint256 agentId, bytes32 jurisdictionHash) external; // ⚠️ verify exact sig
    function setOperator(/* … */) external;                                  // ⚠️ verify exact sig
}
```

**Confidence:** the four bolded query methods + `grantMandate`, `revokeMandate`, `recordExecution`, `freezeAgent` are **high** confidence (verified across rounds). The `⚠️ verify` items appear in the spec's method list but their exact argument lists were not captured in the bundle — read the spec before writing the caller.

### 2.2 `IComplianceProvider` — KYC / eligibility (VAR can wrap World ID here, or pass `address(0)`)

```solidity
interface IComplianceProvider {
    function checkPrincipal(address principal, bytes32 identityRef, bytes32 scopeHash)
        external view returns (bool eligible, ReasonCode reason, uint48 expiresAt);
    function grantPrincipal(/* … */) external;     // ⚠️ verify exact sig
    function revokePrincipal(/* … */) external;    // ⚠️ verify exact sig
}

enum ReasonCode {
    COMPLIANT,             // 0
    KYC_EXPIRED,           // 1
    AML_FLAG,              // 2
    NOT_ACCREDITED,        // 3
    NOT_QUALIFIED,         // 4
    JURISDICTION_BLOCKED,  // 5
    IDENTITY_NOT_FOUND,    // 6
    ATTESTATION_REVOKED,   // 7
    OTHER                  // 8
}
```

`ReasonCode` is the machine-readable failure vocabulary — map these directly to VAR's ERC-1404-style revert reasons and to human-readable UI copy.

### 2.3 Data structures (verified high confidence)

```solidity
struct Mandate {
    address            principal;
    bytes32            identityRef;        // off-chain identity reference (e.g. World ID anchor)
    bytes32            scopeHash;          // keccak256 of off-chain scope doc (IPFS)
    address            complianceProvider; // may be address(0) to skip compliance layer
    MandateScopeParams onChainScope;
    uint48             validFrom;
    uint48             validUntil;
    uint128            cumulativeUsed;     // tracked by recordExecution; does NOT reset on extend
    bool               revoked;
}

struct MandateScopeParams {
    uint128 maxTransactionValue;  // per-tx cap; type(uint128).max = "no limit"
    uint128 maxCumulativeValue;   // lifetime cap
    address assetAddress;         // address(0) = asset-class (cross-asset) mandate
    bytes32 jurisdictionHash;
}
```

**Semantics that bite:**
- **One active principal per `agentId`** — by design, mirrors regulated account segregation. `getActivePrincipal` returns the sole active one.
- `cumulativeUsed` **persists across `extendMandate`**; only revocation resets the relationship.
- `isActiveForAmount` is **VIEW-only** → it checks caps but does NOT mutate `cumulativeUsed`. The cumulative cap is only *enforced for real* when the token calls `recordExecution` (write path). **Skipping `recordExecution` lets the cumulative cap be bypassed.**

## 3. The asset-parameter gap (VAR's single real blocker)

**The finding, stated precisely:** VAR `CONTRACTS.md §5:196` pseudocode calls `isActiveForAmount(agentId, principal, amount, asset)`. The canonical ERC-8226 signature is `isActiveForAmount(agentId, principal, amount)` — **no `asset` parameter**. Asset scope lives in the static `Mandate.onChainScope.assetAddress` field, and asset validation is the **token's** job inside `canTransfer`, not the resolver's.

**Resolution options (pick (a) for the 36h MVP):**

| Option | Conformant? | Cost | Recommendation |
|--------|-------------|------|----------------|
| **(a) Bind EligibilityResolver to a single GatedToken**, drop `asset` param | ✅ yes | trivial | **DO THIS for the demo** — one token, asset is implicit |
| (b) Extend `isActiveForAmount` with an `asset` param | ❌ non-conformant | breaks spec compatibility | avoid |
| (c) Resolve asset via `getMandate().onChainScope.assetAddress` in `canTransfer` before calling `isActiveForAmount` | ✅ yes | +1 storage read | use post-hackathon for multi-asset |

## 4. Integration pattern — the hot path (copy this)

This is the canonical `canTransfer` spine. It is **one VIEW call** per the spec's integration section (no paymaster, no relayer, no off-chain attestation on the hot path → satisfies VAR's "zero new trust assumptions" invariant).

```solidity
// In the ERC-7943-compliant GatedToken:
function canTransfer(address from, address to, uint256 amount)
    external view returns (bool allowed)
{
    uint256 agentId = agentRegistry.getAgentByWallet(from); // ⚠️ verify ERC-8004 ABI (§5)
    if (agentId == 0) return true;                          // non-agent wallet → standard checks

    address principal = IAgentMandate(rams).getActivePrincipal(agentId);
    if (principal == address(0)) return false;              // no active mandate

    // (Option a: single-token resolver → asset check is implicit/optional)
    // (Option c: enforce asset scope before the cap check)
    Mandate memory m = IAgentMandate(rams).getMandate(agentId, principal);
    if (m.onChainScope.assetAddress != address(0) &&
        m.onChainScope.assetAddress != address(this)) {
        return false;                                       // asset not whitelisted in mandate
    }

    return IAgentMandate(rams).isActiveForAmount(agentId, principal, amount);
}
```

**Dual compliance is preserved:** the token's own `canSend`/`canReceive` investor-eligibility checks on `principal` run *in addition to* the RAMS mandate check. Neither bypasses the other.

### Write path (state mutation — NOT in `canTransfer`)

```solidity
// Inside the transfer hook (post-transfer or in-transfer), NOT in the view canTransfer:
IAgentMandate(rams).recordExecution(agentId, principal, amount);
// reverts if amount > maxTransactionValue OR cumulativeUsed + amount > maxCumulativeValue
// caller MUST be the token at onChainScope.assetAddress, a registered RAMS-aware token, or an enforcer
```

### Revocation → "accountability with one click"

```
Human taps "Revoke" on phone
  → IAgentMandate.revokeMandate(agentId, principal)  (PLATFORM tier; sync state write)
  → next block: getActivePrincipal returns address(0) (or mandate.revoked == true)
  → canTransfer returns false
  → agent transfer REVERTS with machine-readable reason
```
This is a **synchronous state change** — the next agent transfer is blocked at the next block, no polling. ⚠️ verify the demo wires the revoke button to the actual on-chain `revokeMandate` (not an off-chain flag) so the block-level guarantee holds.

### Enforcer tiers (freeze)

- `freezeAgent(agentId, jurisdictionHash)`: **global freeze** (`bytes32(0)`) is **REGULATORY tier only**; jurisdiction-scoped freeze allowed for **PLATFORM or REGULATORY**.
- VAR's UI-revoke = **PLATFORM** tier. A regulator-initiated global freeze = **REGULATORY** tier. Both surface on the hot path via the same `isFrozen` check. For MVP: implement PLATFORM revoke only; the enum exists if you want to demo the regulatory tier.

## 5. What this unblocks for the VAR build

- **Lane A (contracts):** `EligibilityResolver` / `AgentBook` lock to the canonical `IAgentMandate`. `grantMandate` param order is pinned (§2.1). `Mandate` + `MandateScopeParams` struct shapes are pinned (§2.3). Asset whitelisting strategy is decided (Option a, §3).
- **Lane B (docs):** cite ERC-8226 as a normative external reference with the "Draft but production-grade" caveat (ERC-8004 launched live in Draft — precedent).
- **Lane C (UI):** AgentBook widget renders mandate state straight from struct fields — `validFrom`, `validUntil`, `cumulativeUsed`, `maxCumulativeValue`, `revoked`. Map `ReasonCode` → human-readable copy.
- **World ID anchor composition:** `identityRef` + `scopeHash` are `bytes32` fields populated at `grantMandate` time. The verified-human anchor flows in via these, and/or via an `IComplianceProvider` wrapper over AgentBook whose `checkPrincipal` returns `eligible` only if the agent wallet maps to the verified human matching `principal`.
- **Dynamic wallet provisioning:** `grantMandate(..., bytes signature)` accepts an off-chain (EIP-712-style) authorization, so Dynamic can sign mandate delegation on the principal's behalf without exposing keys. ⚠️ verify the exact signature scheme (EIP-712 typed data vs. ERC-1271 for smart-contract custody) against the spec's "Signature" section before implementing.

## 6. Adjacent verified facts (carry into other lanes)

- **World ID / AgentBook:** AgentKit (`github.com/worldcoin/agentkit`) is production-available (March 2026). AgentBook registration is **gasless** via hosted relay; default networks World Chain (`eip155:480`) and Base (`eip155:8453`); canonical lookup resolves against the World Chain deployment. `createAgentBookVerifier()` fetches the anonymous human identifier server-side. On-chain `verifyProof(uint256 root, uint256 groupId, uint256 signalHash, uint256 nullifierHash, uint256 externalNullifierHash, uint256[8] proof)` is called **ONCE at seed time** (signal bound to agent wallet, nullifier tracked to prevent replay) — **never on the hot path**.
- **Arc / Circle:** Arc mainnet is **still testnet phase** (June 2026); chain ID `1243`. Testnet USDC `0x3600000000000000000000000000000000000000`. **Dual-decimal trap:** native interface exposes 18 decimals, ERC-20 interface exposes 6 — transfer/approve via the ERC-20 interface only; `MandateScopeParams` value caps are in token base units (USDC = 6 decimals), so size caps accordingly. For the hackathon, prefer Ethereum mainnet (all standards live) and use Arc **testnet** for sponsor-track integration.
- **ERC-8004:** v1 live on Ethereum mainnet (Jan 29, 2026). `getAgentWallet(uint256 agentId) → address`; wallet binding is cleared on agentId transfer. Validation Registry portion still under active update — rely on the **Identity Registry** for VAR.

## 8. Residual open questions

1. **agentId ↔ agent-wallet mapping (HOUR-2, low severity):** ERC-8226 keys on `agentId` (uint256); VAR's `Mandate` (CONTRACTS.md §2:57) stores `agent` (address). Confirm ERC-8004 exposes `getAgentByWallet(address) → uint256`, or store `agentId` on the mandate at grant time. ⚠️ verify the exact ERC-8004 registry ABI before wiring `canTransfer`.
2. **`isActive` / `getMandate` / `isFrozen` / `extendMandate` / `setOperator` exact arg lists** — present in the interface but not captured in the bundle. Read the spec before writing those callers.
3. **`grantMandate` signature scheme** — EIP-712 typed data or ERC-1271? Affects Dynamic integration (§5).
4. **Compliance layer choice:** wrap World ID as an `IComplianceProvider`, or pass `complianceProvider = address(0)` and enforce the World-ID anchor directly in the resolver? Spec permits `address(0)`. Simpler for MVP: `address(0)` + anchor check inline.
5. **`identityRef` encoding from World ID:** `keccak256(worldIdNullifier)` or a structured encoding? Pin the exact derivation so the seed and the check agree.
6. **Asset-whitelist lifecycle:** `onChainScope.assetAddress` is set at `grantMandate` time. Is there an update path, or does changing assets require a new mandate? (Spec shows it static — assume new mandate.)
7. **`AgentOperationDetected` event:** spec says RAMS-aware tokens SHOULD emit it — optional for the demo; add if the dashboard wants a live feed.
8. **Off-chain scope doc format** for `scopeHash` — spec defers JSON format to implementations; define VAR's canonical shape.

## 9. Scope guardrail

ERC-8118 (Agent Authorization) handles *function-level* "what can the agent call" capability delegation — a **separate** standard from ERC-8226's *mandate-level* "who authorized it and with what financial caps." For the VAR MVP, **ERC-8226 mandate compliance is sufficient**; ERC-8118 is out of scope unless the demo needs per-function call permissions.
